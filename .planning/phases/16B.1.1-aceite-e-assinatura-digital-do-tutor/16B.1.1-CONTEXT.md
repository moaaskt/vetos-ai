# Phase 16B.1.1: Aceite e Assinatura Digital do Tutor - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Habilitar o aceite e a assinatura digital (eletrônica) de Termos de Consentimento (`ConsentTerm`) pelo próprio tutor do pet a partir da página pública de visualização do documento (`/documento/:hash`). 

Esta fase abrange:
1. O fluxo no frontend para o tutor ler o documento, marcar o aceite de concordância, informar Nome Completo e CPF e assinar eletronicamente.
2. O registro no banco de dados das evidências de auditoria de aceite (Nome, CPF, IP, User-Agent, Data/Hora).
3. A atualização visual do documento público e do layout de impressão para indicar claramente que o tutor assinou eletronicamente.
4. O envio de uma confirmação ou notificação interna da clínica sobre o aceite realizado.

</domain>

<decisions>
## Implementation Decisions

### Método de Assinatura do Tutor
- **D-01:** O tutor assina eletronicamente através da página pública do documento, acessada pelo link compartilhado (gerado na Phase 16B.1).
- **D-02:** O fluxo do tutor na página pública consiste em:
  1. Visualizar e ler o conteúdo completo do termo.
  2. Marcar uma checkbox obrigatória com o texto confirmando que leu e concorda.
  3. Preencher os campos de Nome Completo e CPF.
  4. Clicar no botão "Aceitar e Assinar" (que realiza uma chamada POST para o backend).
- **D-03:** Não haverá assinatura desenhada em tela nesta fase. O aceite eletrônico com coleta de evidências de auditoria é suficiente.
- **D-04:** O backend deve registrar obrigatoriamente no banco de dados para fins de auditoria:
  - Nome completo informado (`tutorSignatureName`)
  - CPF informado (`tutorSignatureCpf`)
  - Data e hora do aceite (`tutorSignedAt`)
  - Endereço IP do tutor (`tutorSignatureIp`)
  - User-Agent do navegador do tutor (`tutorSignatureUserAgent`)

### Ciclo de Vida e Estados do Documento
- **D-05:** Novos campos no modelo `ConsentTerm` no Prisma schema:
  - `tutorSigned` (Boolean, default `false`)
  - `tutorSignedAt` (DateTime, opcional)
  - `tutorSignatureName` (String, opcional)
  - `tutorSignatureCpf` (String, opcional)
  - `tutorSignatureIp` (String, opcional)
  - `tutorSignatureUserAgent` (String, opcional)
- **D-06:** O documento permanece com o status original da clínica (ex: `SIGNED`), mas com a flag `tutorSigned` marcada como `true` após o fluxo de assinatura.
- **D-07:** O hash SHA-256 e o QR Code originais permanecem válidos e inalterados. O mesmo hash continua sendo usado para acessar a página pública.
- **D-08:** Validação de Integridade: Um termo só pode ser assinado pelo tutor se já tiver sido assinado e registrado previamente pela clínica (status `SIGNED`). O backend deve validar e recusar assinaturas em documentos com status `DRAFT`.

### Apresentação Visual e Impressão
- **D-09:** Se o documento já estiver assinado pelo tutor (`tutorSigned === true`), a página pública oculta o formulário de assinatura e passa a exibir uma seção de auditoria com os dados do aceite (Nome, CPF, IP, data/hora formatada localmente).
- **D-10:** A seção de auditoria com a assinatura/aceite do tutor deve ser exibida no final do layout do Termo de Consentimento, tanto na tela pública quanto no layout de impressão física (`window.print()`).
- **D-11:** O cabeçalho de autenticação superior deve ser atualizado quando houver assinatura do tutor para deixar explícito que ambas as partes (clínica e tutor) assinaram/validaram o documento.

### Confirmações e Notificações
- **D-12:** Após o aceite ser concluído, o status visual na timeline de prontuário do pet no `PetDetails.tsx` deve atualizar para refletir que o termo de consentimento foi assinado pelo tutor.
- **D-13:** Registrar no `NotificationLog` (ou gerar uma notificação in-app/registro interno) informando a clínica sobre a conclusão da assinatura.

### the agent's Discretion
- Design visual do formulário de aceite (checkbox, campos de input Nome/CPF, botão) na página pública `/documento/:hash`.
- Design da seção de evidências de assinatura do tutor no final do documento para impressão e tela.
- Nome e implementação exatos dos novos endpoints do backend para registrar o aceite (ex: `POST /consent-terms/:hash/tutor-sign`).
- A lógica de sanitização de CPF no backend.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `frontend/src/pages/PublicDocument.tsx` (ou similar criado na Phase 16B.1): Página standalone para visualização pública do documento pelo hash.
- `backend/src/consent-terms/consent-terms.service.ts` e `backend/src/consent-terms/consent-terms.controller.ts`: Lógica existente de manipulação de termos de consentimento.

### Established Patterns
- Roteamento público sem autenticação para rotas `/documento/:hash` e `/verify/:hash`.
- Captura de IP do request no NestJS via `@Ip() ip: string` ou lendo headers `x-forwarded-for` de proxies reversos.
- Captura de User-Agent via `@Headers('user-agent')` no controller do NestJS.

### Integration Points
- **PublicDocument.tsx (Frontend)**: Integrar o formulário de preenchimento de Nome/CPF e checkbox de concordância na base do termo de consentimento se o status for `SIGNED` mas `tutorSigned` for falso. Exibir o bloco de auditoria se `tutorSigned` for verdadeiro.
- **consent-terms.controller.ts & service.ts (Backend)**: Novo endpoint público (sem token JWT) ex: `POST /consent-terms/:hash/tutor-sign` recebendo Nome, CPF, IP e User-Agent para registrar o aceite se o hash for válido e o status for `SIGNED`.

</code_context>

<specifics>
## Specific Ideas

- O formulário de assinatura do tutor deve ser intuitivo e adaptado para telas de celulares, visto que a maioria dos tutores acessa links de WhatsApp por dispositivos móveis.
- Exibir uma mensagem de sucesso elegante após o clique em "Aceitar e Assinar", permitindo que o tutor imprima ou salve a versão com os dados da assinatura se desejar.

</specifics>

<deferred>
## Deferred Ideas

- **Assinatura Desenhada na Tela (Canvas)**: Implementação de canvas para desenhar com dedo/mouse — fora do escopo desta fase por decisão explícita de focar em aceite eletrônico.
- **Autenticação OTP (One-Time Password)**: Envio de código via e-mail ou WhatsApp para validar o telefone/e-mail do tutor antes de permitir a assinatura — pode ser adicionado futuramente para maior conformidade legal.

</deferred>
