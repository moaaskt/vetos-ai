# Phase 16B.1: Compartilhamento de Documentos com Tutor - Research

**Gathered:** 2026-06-18  
**Status:** Completed Research  

---

## 1. Background & Objectives

O objetivo desta pesquisa é estruturar a integração do fluxo de assinatura digital de documentos clínicos (Receitas Médicas e Termos de Consentimento) da Phase 16B com a infraestrutura de notificações de Email (SMTP) e WhatsApp (Evolution API) da Phase 4. O compartilhamento será uma ação manual e explícita acionada pelo veterinário, enviando uma mensagem contextualizada com o link para visualização pública e sem autenticação del documento assinado (`/documento/:hash`).

---

## 2. Codebase Investigation Findings

### 2.1. Infraestrutura de Notificações (SMTP & Evolution API)
- **NotificationsProcessor & BullMQ:** O processador `NotificationsProcessor` em `backend/src/notifications/notifications.processor.ts` está associado à fila `'notifications'` e gerencia o envio assíncrono de notificações de acordo com o canal (`EMAIL` ou `WHATSAPP`).
- **Envio por Canal:**
  - **EMAIL:** Roteia para o `SmtpProvider`, que busca dinamicamente as credenciais SMTP criptografadas em `NotificationConfig` correspondente à clínica, realiza o decryption da senha e faz o disparo via `nodemailer`.
  - **WHATSAPP:** Se houver configuração válida da Evolution API habilitada no `NotificationConfig` da clínica, delega o envio ao `EvolutionApiProvider`. Este por sua vez faz uma requisição HTTP POST para a API Evolution (`/message/sendText/:instanceName`) autenticando via API Key da clínica. Caso contrário, faz fallback para o `WhatsAppMockProvider` (ou mock direto).
- **Disparo de Mensagens:** Para enfileirar novas notificações a partir dos fluxos de receitas e termos de consentimento, deve-se injetar o `NotificationsService` e chamar o método `enqueueNotification`. O payload deve ter o tipo `EnqueueNotificationInput` com o evento correspondente (`PRESCRIPTION_SHARED` ou `CONSENT_TERM_SHARED`).

### 2.2. Modelos Prisma (schema.prisma) & Rastreabilidade
Para suportar o rastreamento simplificado na timeline e o histórico detalhado de auditoria de notificações, o `schema.prisma` deve ser enriquecido da seguinte forma:

1. **lastSharedAt em Documentos:**
   - Adicionar o campo `lastSharedAt DateTime?` nos modelos `Prescription` e `ConsentTerm`. Este campo será atualizado sempre que um compartilhamento for concluído com sucesso.

2. **Novos relacionamentos em NotificationLog:**
   - Para garantir a rastreabilidade direta do envio ao documento compartilhado, adicionar os campos opcionais `prescriptionId` e `consentTermId` no modelo `NotificationLog`:
     ```prisma
     model NotificationLog {
       // ... campos existentes ...
       prescriptionId    String?
       consentTermId     String?
       
       prescription      Prescription?   @relation(fields: [prescriptionId], references: [id], onDelete: SetNull)
       consentTerm       ConsentTerm?    @relation(fields: [consentTermId], references: [id], onDelete: SetNull)
       
       // Adicionar índices para performance de queries
       @@index([prescriptionId])
       @@index([consentTermId])
     }
     ```
   - No `NotificationsProcessor`, atualizar o método `createNotificationLog` para receber esses IDs no payload `EnqueueNotificationInput` e salvá-los no log.

### 2.3. Rota Pública no Frontend & Endpoint Público no Backend
- **Backend (Novo Endpoint Público):**
  - O `VerificationController` (em `backend/src/verification/verification.controller.ts`) já está exposto publicamente sem proteção de rota (sem `JwtAuthGuard`).
  - Para servir a visualização do documento completo, deve-se expandir este controller adicionando um novo endpoint público: `GET /verify/:hash/details` (ou `GET /verify/:hash/public`).
  - Este endpoint deve realizar a busca no Prisma pelo `documentHash === hash` e `status === SIGNED` tanto na tabela `Prescription` quanto em `ConsentTerm`. Se localizado, retorna o payload clínico completo (incluindo medicamento, dosagem, frequência, etc. para receitas, ou texto final estruturado para termos de consentimento), além de relacionamentos necessários como dados do paciente (Pet) e da clínica (Clinic) para a impressão.
- **Frontend (Rota Pública Standalone):**
  - Adicionar a rota pública `/documento/:hash` no React Router.
  - Desenvolver o componente `PublicDocumentView.tsx` de forma standalone (sem renderizar a Sidebar ou Topbar do dashboard da clínica).
  - Este componente consome o endpoint `GET /verify/:hash/details` do backend passando o hash obtido via `useParams`.
  - Se os dados forem válidos, exibe uma barra superior simplificada com botão "Imprimir" (chamando `window.print()`) e reutiliza diretamente os componentes de layout de impressão existentes: `PrintReceita` ou `PrintTermo`.
  - Tratamento de loading states e visualização limpa de erros (como "Documento inválido ou expirado").

### 2.4. Integração do Botão "Enviar ao Tutor" na UI do Frontend
O botão para compartilhamento deve estar disponível em dois locais fundamentais:
1. **Modal de Preview de Impressão (`PrintPreviewModal.tsx`):**
   - Adicionar o botão "Enviar ao Tutor" ao lado do botão "Imprimir" quando `isSigned` for verdadeiro (`status === 'SIGNED'`) e `type !== 'prontuario'`.
2. **Timeline de Prontuário (`PetDetails.tsx`):**
   - Para os itens do tipo `PRESCRIPTION` e `CONSENT_TERM` cujo `status === 'SIGNED'`, adicionar a opção "Enviar ao Tutor" ou ícone de compartilhamento ao lado do botão de visualizar impressão.
- **Fluxo de Seleção de Canais:**
  - Ao clicar no botão de compartilhamento, exibir um modal (`ShareDocumentModal.tsx`) para seleção de canais.
  - O modal carrega os dados de contato do tutor (Email e Telefone/WhatsApp).
  - Checkboxes para Email e WhatsApp ficam habilitados ou desabilitados com tooltips informativos dependendo da existência ou ausência dos dados de contato no cadastro do tutor.
  - Ao submeter, faz chamada aos novos endpoints autenticados de compartilhamento no backend: `POST /prescriptions/:id/share` e `POST /consent-terms/:id/share`.

## 3. Validation Architecture

### 3.1. Testes Automatizados

#### 3.1.1. Testes Unitários (Backend Services)
- **PrescriptionsService & ConsentTermsService:**
  - Testar que a tentativa de compartilhamento de documentos com status `DRAFT` lança erro `BadRequestException` ("Apenas documentos assinados podem ser compartilhados").
  - Testar que, ao chamar a função de compartilhamento de um documento assinado, o `NotificationsService.enqueueNotification` é chamado com o payload correto de acordo com os canais selecionados.
  - Testar que `lastSharedAt` é devidamente atualizado no banco de dados para a data atual pós-disparo.

#### 3.1.2. Testes de Integração (Public Endpoint & Database)
- **VerificationController (Público):**
  - Validar que a rota `/verify/:hash/details` retorna com sucesso os dados estruturados do documento assinado correspondente ao hash, e que não requer cabeçalhos de autenticação (JWT).
  - Validar que a rota retorna `404 Not Found` caso o hash não corresponda a nenhum documento assinado ativo.
- **NotificationLog Audit Trace:**
  - Inserir um job de notificação de compartilhamento e validar no banco de dados que as chaves estrangeiras (`prescriptionId` ou `consentTermId`) são salvas de forma correta no `NotificationLog`.

### 3.2. Testes Manuais (E2E & Fluxos Visuais)

- **Cenário 1: Cadastro Incompleto do Tutor**
  - Acessar o modal de compartilhamento para um documento assinado de um pet cujo tutor não possui email cadastrado.
  - Verificar se a checkbox de Email está desabilitada com um tooltip explicativo ("Tutor não possui e-mail cadastrado"). O mesmo deve valer para o WhatsApp se não houver telefone.

- **Cenário 3: Envio de Notificações com Sucesso**
  - Realizar o envio para Email e WhatsApp de uma receita assinada.
  - Confirmar a exibição do toast de sucesso no frontend.
  - Verificar se o log correspondente foi gravado no banco com status `SENT` e se o campo `lastSharedAt` do documento foi populado.
  - Verificar no mailcatcher ou log de console (ou WhatsApp Mock) a entrega da mensagem contextualizada com o link correto `/documento/:hash`.

- **Cenário 4: Acesso ao Link Público**
  - Copiar o link público enviado na notificação e abri-lo em uma janela anônima (sem cookies ou tokens de autenticação).
  - Validar se a página carrega corretamente sem o layout do app (sidebar/topbar), exibindo o documento em alta fidelidade e permitindo a impressão via `window.print()` ao acionar o botão de impressão.
