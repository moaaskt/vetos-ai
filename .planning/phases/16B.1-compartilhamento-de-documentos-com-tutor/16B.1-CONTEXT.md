# Phase 16B.1: Compartilhamento de Documentos com Tutor - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Permitir o envio de documentos clínicos assinados (Receitas Médicas e Termos de Consentimento) para o tutor do pet via Email e WhatsApp. Esta fase integra:
1. O fluxo de assinatura digital da Phase 16B (modelos `Prescription` e `ConsentTerm` com status `SIGNED`, hash SHA-256, QR code de verificação).
2. A infraestrutura de notificações da Phase 4 (`SmtpProvider` para email SMTP por clínica, `EvolutionApiProvider` para WhatsApp via Evolution API, `NotificationLog` para auditoria).
3. Uma página pública de visualização do documento assinado acessível ao tutor sem login.

</domain>

<decisions>
## Implementation Decisions

### Gatilho de Envio
- **D-01:** O compartilhamento é uma ação **manual e explícita** — assinar e compartilhar são operações distintas. O veterinário clica "Enviar ao Tutor" após a assinatura do documento.
- **D-02:** O botão "Enviar ao Tutor" deve estar presente em **dois locais**:
  - No **modal fullscreen de preview** do documento assinado, ao lado do botão "Imprimir" (fluxo imediato: Assinar → Visualizar → Enviar ao Tutor).
  - No **dropdown de ações da timeline** do `PetDetails.tsx`, visível apenas para documentos com status `SIGNED` (permite reenvio posterior).

### Formato do Documento Enviado
- **D-03:** O envio **não gera PDF server-side** nesta fase. O email/WhatsApp contém uma mensagem contextualizada com um **link para uma página pública de visualização rica** (`/documento/:hash`).
- **D-04:** A página pública de visualização (`/documento/:hash`) deve:
  - Ser uma rota pública no React Router, **sem exigir login**.
  - Não renderizar sidebar/navbar do app (layout standalone limpo).
  - Buscar os dados públicos do documento pelo hash via endpoint backend.
  - **Reutilizar os componentes de layout de impressão** da Phase 16B (`PrintReceita.tsx`, `PrintTermo.tsx`).
  - Exibir o documento completo assinado com status de assinatura, hash, data e QR code.
  - Permitir impressão via `window.print()`.
- **D-05:** O endpoint backend `/verify/:hash` existente (Phase 16B D-14) deve ser **expandido** para retornar os dados completos do documento (não apenas resumo) quando acessado pela página pública. Alternativa: criar um novo endpoint dedicado (ex: `GET /documents/:hash/public`) que retorna o conteúdo completo para renderização.

### Canais e Preferência
- **D-06:** O veterinário **escolhe os canais no momento do envio** através de um modal de seleção contendo:
  - Nome do tutor.
  - Email cadastrado (checkbox habilitado se existir, desabilitado com tooltip se não).
  - Telefone/WhatsApp cadastrado (checkbox habilitado se existir, desabilitado se não).
  - Botão "Enviar".
- **D-07:** Após o envio, exibir **feedback claro** (toast/notificação) informando quais canais foram utilizados e o resultado (sucesso/falha) por canal.
- **D-08:** **Não enviar automaticamente** para todos os canais — o veterinário deve ter controle explícito dado a relevância jurídica dos documentos.

### Rastreabilidade e Auditoria
- **D-09:** Abordagem **híbrida** para rastreabilidade:
  - **`NotificationLog`**: Registrar cada tentativa de envio com event type `PRESCRIPTION_SHARED` ou `CONSENT_TERM_SHARED`. Campos utilizados: `clinicId`, `petId`, `channel`, `to`, `status` (`SENT`/`FAILED`), `event`, `errorMessage`. Adicionar campo opcional `documentId` (ou `prescriptionId`/`consentTermId`) ao modelo `NotificationLog` para referência direta ao documento compartilhado.
  - **`lastSharedAt` (DateTime?)**: Adicionar campo nos modelos `Prescription` e `ConsentTerm` para acesso rápido à última data de compartilhamento (exibição na timeline/UI sem joins pesados).
- **D-10:** O `NotificationLog` preserva auditoria completa por envio (canal, destinatário, sucesso/falha, data, reenvios). O campo `lastSharedAt` é atualizado a cada envio bem-sucedido.

### the agent's Discretion
- Conteúdo e template da mensagem contextualizada enviada por Email e WhatsApp (texto informativo com link do documento).
- Componente visual do modal de seleção de canais (reutilizar `Modal.tsx` existente ou popover).
- Decisão sobre expandir `/verify/:hash` vs criar novo endpoint `/documents/:hash/public`.
- Estilização da página pública `/documento/:hash` (layout standalone, cores, responsividade).
- Estratégia de loading/error states na página pública (documento não encontrado, hash inválido).

</decisions>

## Existing Code Insights

### Reusable Assets
- `frontend/src/pages/PetDetails.tsx`: Timeline clínica com ícones por tipo de documento (📝 💊 📄 📎) — ponto de integração do dropdown "Enviar ao Tutor" para documentos SIGNED.
- `frontend/src/components/Modal.tsx`: Modal reutilizável — base para o modal de seleção de canais e para o modal de preview.
- `backend/src/notifications/notifications.processor.ts`: Processor BullMQ já configurado com roteamento EMAIL vs WHATSAPP, criação de `NotificationLog`, validação de contato. **Pode ser reutilizado diretamente** adicionando suporte ao novo event type.
- `backend/src/prescriptions/prescriptions.service.ts`: Método `sign()` com geração de hash SHA-256, QR code e campo `verificationUrl`. **O hash e a URL de verificação já existem** — a página pública consome esses dados.
- `backend/src/notifications/providers/smtp.provider.ts`: Provider SMTP com carregamento de credenciais por clínica e envio via Nodemailer.
- `backend/src/notifications/providers/evolution-api.provider.ts`: Provider WhatsApp com integração HTTP à Evolution API.

### Established Patterns
- Guards `JwtAuthGuard` + `@CurrentUser()` em todos os endpoints autenticados — endpoints de compartilhamento seguem este padrão.
- Validação de pertença do Pet à clínica antes de qualquer operação (replicar no endpoint de compartilhamento).
- Roteamento de canal no `NotificationsProcessor.send()`: EMAIL → `SmtpProvider`, WHATSAPP → `EvolutionApiProvider` (com fallback para mock).
- Enum `DocumentStatus` (DRAFT | SIGNED) usado em Prescription e ConsentTerm.

### Integration Points
- **Modal de preview (Phase 16B)**: Adicionar botão "Enviar ao Tutor" ao lado de "Imprimir" quando `status === SIGNED`.
- **Dropdown de ações na timeline**: Adicionar opção "Enviar ao Tutor" para documentos SIGNED.
- **NotificationsProcessor**: Adicionar suporte a novos events (`PRESCRIPTION_SHARED`, `CONSENT_TERM_SHARED`) com construção de mensagem contextualizada contendo link `/documento/:hash`.
- **React Router**: Adicionar rota pública `/documento/:hash` sem autenticação.
- **Backend endpoint**: Expandir ou criar endpoint público para servir dados completos do documento por hash.

<specifics>
## Specific Ideas

- O veterinário deve ter controle explícito sobre o compartilhamento dado a relevância jurídica dos documentos (receitas e termos assinados).
- A página pública `/documento/:hash` deve reutilizar os componentes `PrintReceita.tsx` e `PrintTermo.tsx` da Phase 16B, mantendo consistência visual.
- A mensagem enviada por email/WhatsApp deve ser contextualizada: "Dr(a). [nome] compartilhou [tipo do documento] do paciente [nome do pet]. Acesse: [link]".
- Discovery completo da infraestrutura existente de SMTP e Evolution API deve ser feito pelo researcher antes do planejamento, para validar que os providers estão funcionais.
- O hash SHA-256 já é calculado no `sign()` do PrescriptionsService — o link `/documento/:hash` consome esse hash diretamente.

</specifics>

<deferred>
## Deferred Ideas

- **Geração de PDF server-side**: Gerar PDF com Puppeteer/Playwright no backend para anexar ao email — escopo de fase futura (infra pesada).
- **Envio automático ao assinar**: Disparar compartilhamento automaticamente quando o documento for assinado — pode ser adicionado como configuração opcional futura.
- **Envio de anexos de exames (Phase 16A)**: Compartilhar laudos e imagens da aba "Exames e Anexos" com o tutor — escopo separado.
- **Histórico de compartilhamentos visível na UI**: Painel com listagem de todos os envios feitos por documento — pode usar o NotificationLog, mas a tela de visualização é escopo futuro.

</deferred>
